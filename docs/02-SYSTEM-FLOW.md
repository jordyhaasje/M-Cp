# System Flow (Runtime & Integratie)
Doelgroep: repo maintainers en AI-agents.

Dit document beschrijft hoe requests, authenticatie, theme editing en store-integraties op de actuele codebase functioneren.

## 1. Onboarding & Credentials
1. Een gebruiker registreert zich op de License Service en koppelt Shopify primair via een Shopify Admin API access token (`shopAccessToken`) uit een merchant-created custom app.
2. `shopClientId` / `shopClientSecret` blijft ondersteund voor trusted app-achtige setups, maar is niet de standaardinstructie voor merchant-created custom apps.
3. Validatie controleert op `REQUIRED_SHOPIFY_ADMIN_SCOPES`, inclusief `read_themes`, `write_themes`, `read_fulfillments`, `read_merchant_managed_fulfillment_orders` en `write_merchant_managed_fulfillment_orders`.
4. Theme file writes kunnen alsnog door Shopify worden geweigerd wanneer de app geen theme file write access/exemption heeft. De MCP geeft dan `theme_write_exemption_required` terug.
5. Credentials en secrets blijven server-side op de License Service en worden nooit aan de MCP client doorgegeven.

## 2. Authenticatie: OAuth PKCE & Fallback
Vanaf 2026 ondersteunt Hazify native OAuth 2.0 PKCE.
1. **Discovery:** Client ontdekt auth servers via `/.well-known/oauth-protected-resource`.
2. **Authorize flow:** `GET /oauth/authorize` levert de login pagina; `POST /oauth/authorize` handelt toestemming of weigering af.
3. **Token:** `/oauth/token` wisselt de autorisatiecode in voor access tokens.

Belangrijk voor compat-clients:
- De authorize-roundtrip canonicaliseert semantisch gelijke MCP scopes (`mcp:tools` versus `mcp:tools:read mcp:tools:write`) en bewaart toegestane auxiliary scopes zoals `offline_access`. Daardoor falen clients niet meer op pure scope-volgorde of query/form echo-verschillen.
- Toegestane MCP scopes zijn strikt: `mcp:tools`, `mcp:tools:read` en `mcp:tools:write`. Onbekende `mcp:*` tokens zoals `mcp:admin` falen dicht en krijgen geen fallback naar full access.
- Tokens worden aan de MCP resource URL gebonden. De remote vergelijkt `resource`, `targetResource` of `aud` uit introspectie met de eigen publieke `/mcp` resource en weigert mismatch.

Fallback: dashboard-tokens blijven bruikbaar voor pure API of legacy connectors.

## 3. Remote MCP Request Lifecycle (`/mcp`)
De service in `apps/hazify-mcp-remote/src/index.js` luistert alleen op Streamable HTTP.
1. Een request bevat een token via `Authorization: Bearer` of `x-api-key`.
2. Indien aanwezig wordt `Origin` gevalideerd tegen de allowlist.
3. De remote doet token-introspectie via de License Service en controleert scope, resource-binding, licentie, tool-entitlements en mutation scopes.
   - Tool-entitlements worden gecontroleerd op zowel de aangevraagde aliasnaam als de canonieke toolnaam. Een geblokkeerde `set-order-tracking` blokkeert dus ook aliases zoals `add-tracking-to-order` en `update-order-tracking`.
4. Shopify token exchange gebeurt **lazy**: pas vlak voor een toolcall die Shopify echt nodig heeft.
5. Theme draft persistence en file locks lopen via PostgreSQL; er is geen runtime fallback naar in-memory opslag.
6. `initialize` en `tools/list` blijven daardoor snel en contextvrij.

## 5. Theme Editing Pipeline
- Canonical agent flow: gebruik `get-themes` alleen voor discovery wanneer de gebruiker nog geen expliciet `themeId` of veilig `themeRole` heeft gegeven; daarna `plan-theme-edit` -> compacte read/preflight -> `create-theme-section` voor nieuwe sections -> optioneel aparte `mode="edit"` of `patch-theme-file` voor vervolgfixes.
- `themeRole` zonder `themeId` is alleen veilig voor `main`, omdat Shopify alleen `MAIN` als uniek garandeert. Voor `development`, `unpublished` en `demo` is een exact `themeId` verplicht.
- Bestaande theme edit: `search-theme-files` -> `get-theme-file` -> `draft-theme-artifact` of `patch-theme-file`.
- Nieuwe section creates lopen bij voorkeur via `create-theme-section`; generieke multi-file of advanced theme writes blijven via `draft-theme-artifact` lopen.
1. Start voor native product-blocks, theme blocks en template placement eerst met `plan-theme-edit` op hetzelfde expliciete theme. Die planner geeft compact terug of je een single-file patch, multi-file edit of create-section flow nodig hebt. Voor `new_section` retourneert de planner nu naast compacte `themeContext`-guardrails ook een `sectionBlueprint` met section-category metadata, exacte required reads, helper snippets, parser/media/theme guardrails én een generiek `implementationContract` voor schema, rendering, media, scoped interactiviteit en preserve-on-edit discipline. Screenshot-only exacte replica's zonder losse bron-assets krijgen daarbij aparte reference-signals zoals `previewMediaPolicy="best_effort_demo_media"` en `allowStylizedPreviewFallbacks=true`, zodat de write-flow de layout/styling exact kan mikken zonder te doen alsof losse bronmedia al beschikbaar zijn.
2. `plan-theme-edit` beschrijft nu de directe volgende stap via `nextTool` en `nextArgsTemplate`. Als er eerst theme-context gelezen moet worden, wijst de planner dus bewust eerst naar `get-theme-file` of `get-theme-files`; de uiteindelijke write-tool staat apart in `writeTool` en `writeArgsTemplate`.
3. Lees voor `new_section` en native block flows exact de planner-provided `nextReadKeys` met `includeContent=true` in. Daardoor ziet de LLM vooraf de relevante section/snippet helpers, wrappers en renderer-architectuur zonder brede wildcard reads te doen. Voor follow-up prompts op een net aangemaakte of exact geplande section kan dezelfde file sticky blijven zolang het expliciete theme-target compatibel blijft. Een expliciete theme-switch verbreekt die koppeling; alleen `themeRole='main'` mag nog veilig aan een eerder bevestigd `themeId` blijven hangen omdat Shopify daar maar één live theme tegelijk toestaat.
4. Lees daarna met `get-theme-file` alleen het exacte doelbestand in dat je gaat aanpassen. Voor native product-blocks is dat meestal de bestaande section en eventuele snippet renderer; herlees het planner-geselecteerde templatebestand alleen als placement expliciet gevraagd is. `patch-theme-file` vereist nu eerst zo’n exacte read en weigert generieke of niet-unieke anchors vroegtijdig.
5. Voer nieuwe losse section-creaties bij voorkeur uit via `create-theme-section`. Gebruik `draft-theme-artifact` in `mode="edit"` voor bestaande bestanden en in `mode="create"` voor advanced create-flows of multi-file writes. Gebruik bij voorkeur `patch` voor één gerichte wijziging of `patches` voor meerdere sequentiële wijzigingen in hetzelfde bestand om token-verbruik te beperken en truncation te voorkomen. De pipeline valideert section/block schema-ranges nu ook vooraf op bounds, step-alignment, kleine discrete ranges en de Hazify preflight-guard van maximaal 101 stappen. Bij een foutieve range-default geeft de tool nu ook exacte `suggestedReplacement.default` en geldige candidate-defaults terug zodat een retry minder hoeft te gokken.
6. Als `patch-theme-file` `patch_scope_too_large` teruggeeft, volg exact de repair response. Met `readContext.currentReadContextValid=true` wijst de server direct naar `draft-theme-artifact mode="edit"` met dezelfde structurele patch/patches en `baseChecksumMd5`. Alleen wanneer `requiresReread=true` moet de client eerst `get-theme-file includeContent=true` uitvoeren.
7. Full rewrites in `mode="edit"` moeten alles behouden tenzij expliciet gewijzigd. De preflight blokkeert compacte reconstructies, context-placeholders en lossy rewrites die duidelijk kleiner zijn dan de actuele file. Ook verlies van bestaande schema-settings, block types/settings, presets, `block.shopify_attributes`, `image_tag`-renderpaden, section.id CSS-scoping en Impact/theme helpers leidt tot `inspection_failed_lossy_rewrite` of een specifieker inspectie-error.
8. `create-theme-section` leidt intern nogmaals compacte theme-context en section-blueprint metadata af voordat de daadwerkelijke guarded create-flow draait. Daardoor kan dezelfde request niet alleen schema/lint-fouten afvangen, maar ook hero-achtige oversizing van typography, spacing, gaps en min-heights blokkeren wanneer die duidelijk buiten de content-conventies van het doeltheme vallen. Daarnaast weigert de tool create-writes nu zolang verplichte planner-reads nog niet echt zijn uitgevoerd. Voor screenshot-only exacte replica's zonder losse bron-assets mag de guarded flow wel renderbare demo-media of een gestileerde media shell toestaan, zolang de layout/styling en merchant settings trouw aan de referentie blijven.
9. Lokale preflight-validatie bundelt nu meerdere deterministische fouten in één response met machine-readable velden zoals `errorCode`, `errors[]`, `lintIssues[]`, `normalizedArgs`, `nextAction`, `retryMode`, `suggestedSchemaRewrites`, `preferSelectFor` en waar relevant `themeContext` en `sectionBlueprint`. Operationele blockers zoals patch-anchor fouten, mixed create/edit zonder mode, read failures en preview write failures blijven fail-fast. Shopify richtext-default fouten worden explicieter teruggegeven via codes zoals `richtext_default_forbidden_tag`; te minimale premium standalone sections geven `standalone_section_too_minimal`. Parser-onveilige `${...{{ ... }}...}` combinaties, geneste `{{ ... }}` binnen dezelfde output-tag en niet-gescopede interactieve JS worden nu ook vóór preview-write gevangen. Optionele resource-settings zoals `image_picker`, `video` en `video_url` moeten bovendien blank-safe worden gerenderd in zowel create- als editflows: onbeschermde `image_url`, `video_tag` of `external_video_*` chains op lege merchant-settings worden nu als preflight-fout afgekeurd. Presets met blocks zonder ingevulde media mogen daardoor niet meer crashen in de Theme Editor. Ontbrekende verplichte schema-velden zoals setting/block `label`, `type`, `id`, `name` en `content` waar relevant falen nu ook al in deze lokale inspectie in plaats van pas later in `theme-check`. Select-defaults moeten in `options` bestaan, loops over `section.blocks` moeten `block.shopify_attributes` binnen dezelfde loop-body renderen, en Impact-like contexts moeten correcte wrappers/scoped CSS gebruiken. Als zowel lokale inspectie als lint stuk zijn, wordt dat nu in dezelfde preflight-response geretourneerd. Wanneer `plannerHandoff` aanwezig is, gebruikt deze preflight ook de planner-afgeleide context en blueprint zodat stateless clients minder context verliezen.
10. Live of ander target toepassen gebeurt via `apply-theme-draft` met expliciete confirmation. Deze tool is nooit de eerste write-stap voor een nieuwe section. Opgeslagen `theme_drafts` zijn aan `shop_domain` gebonden; apply naar een andere Shopify shop faalt met `theme_draft_shop_mismatch`.
11. Verifieer writes via de `verify` output van `draft-theme-artifact` of aanvullend met `verify-theme-files` of `get-theme-file`. Preview/apply is pas succesvol wanneer verify-after-write geen `mismatch`, `missing` of verify-error bevat.
12. Als een gebruiker placement op homepage/producttemplate expliciet vraagt, doe dat als aparte `mode="edit"` wijziging op het juiste `templates/*.json` of `templates/*.liquid` bestand van hetzelfde gekozen theme.

## 6. Shopify-conforme File Policy
- Beperk writes tot de noodzakelijke theme-bestanden; voor section-wijzigingen is dat meestal `sections/<handle>.liquid`.
- Nieuwe sections moeten naast een geldig `{% schema %}` block ook renderbare markup bevatten; schema-only of style-only stubs worden geblokkeerd.
- Optionele merchant-editable resources moeten blank-safe worden gerenderd. Gebruik dus guards of expliciete defaults/fallbacks voordat je `image_url`, `video_tag`, `external_video_url` of vergelijkbare resource-filters aanroept.
- `richtext.default` moet Shopify-veilige HTML gebruiken: top-level `<p>` of `<ul>`, zonder niet-toegestane tags zoals `<mark>`.
- Voeg alleen `snippets/` toe als markup of logica echt herhaald wordt.
- Voeg alleen `blocks/` files toe als je bewust theme blocks nodig hebt.
- Voeg alleen `locales/*.json` toe bij vaste, niet-merchant-editable UI strings.
- Maak geen assets standaard; gebruik component-scoped CSS/JS in de section zelf.
- Geen Liquid binnen `{% stylesheet %}` of `{% javascript %}`.
- `templates/*.json` en `config/*.json` writes zijn alleen toegestaan in `mode="edit"` en worden onderworpen aan JSON/JSONC-validatie (met controle op verplichte `sections`/`order`/`current` datastructuren). `templates/*.liquid` writes zijn in `mode="edit"` ook toegestaan, maar krijgen Liquid safety-inspectie in plaats van JSON-validatie. In `mode="create"` blijven templates/config verboden.
- Merchant placement van nieuwe secties blijft bij voorkeur via de Theme Editor plaatsvinden indien er geen bestaande flow is.

## 7. Veiligheid, Rate Limiting & Tool Hardening
1. **Tenant isolation:** Toolcalls blijven strikt aan de juiste shop en tenant gekoppeld.
2. **Tool hardening:** Destructieve of financiële mutaties vereisen expliciete `confirmation` strings en vaak ook `reason`.
3. **Core file protection:** `layout/theme.liquid` mag `content_for_header` en `content_for_layout` niet verliezen.
4. **Truncation & Lossy Rewrite Protection:** In `mode="edit"` blokkeert de pipeline automatisch schrijfacties die leiden tot >50% bestandverkleining, risicovolle >25% krimp zonder expliciete verwijder-intentie, verlies van kritieke JSON-velden (zoals `sections` in templates) of verlies van bestaande section-contracten zoals schema-settings, block settings, presets, `block.shopify_attributes`, `image_tag`-paden, scoped CSS en theme wrappers.
5. **Linter Resilience:** De `theme-check` linter is zo afgesteld dat externe ontbrekende snippet-fouten de write niet blokkeren, maar als warnings worden weergegeven. Echte Liquid-syntaxfouten blijven wel hard falen en worden nu check-specifiek teruggegeven met line/column metadata en repair hints.
6. **Distributed locks:** Theme writes gebruiken PostgreSQL advisory locks om parallelle conflicts te voorkomen.
